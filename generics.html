30.1

Introduction to Multithreading

What is Multithreading?

Imagine you're using a word processor. You're typing, and at the same time, the app is auto-saving your file and also checking spelling in the background. All three things are happening at the same time. That's multithreading.

Definition

Multithreading = Running multiple tasks concurrently inside one program. Java has built-in support — no external libraries needed.

Why Do We Need It?

Without MultithreadingWith Multithreading
Tasks run one by oneTasks run at the same time
Program feels slow / frozenProgram feels fast and responsive
CPU sits idle oftenCPU is used more efficiently
🍳
Real-Life Analogy

Think of a restaurant kitchen: one cook handles salads, another handles the main course, another handles desserts — all at the same time. That's multithreading: multiple "cooks" (threads) working on different "dishes" (tasks) simultaneously.

💡 Tips & Tricks — 30.1

Exam Tip: Multithreading is needed because the CPU is often idle (waiting for input, file I/O, etc.). Threads fill that idle time with useful work.

Common Mistake: Students think multithreading always requires multiple CPUs. No! Even on a single CPU, multithreading works via time-sharing — the OS switches between threads so fast it feels simultaneous.

30.2

Thread Concepts

What is a Thread?

A thread is like a single path of execution — a sequence of instructions from start to finish. Think of it like a lane on a highway: multiple lanes = multiple threads; cars (instructions) move through their lane independently.

Multi-CPU vs Single-CPU

Multi-CPU (Parallel)Single-CPU (Time Sharing)
CPU 1 → Thread 1 runs
CPU 2 → Thread 2 runs
CPU 3 → Thread 3 runs
All truly at the same time!		 OS rapidly switches:
Slice 1 → Thread 1
Slice 2 → Thread 2
Slice 3 → Thread 3...
Appears simultaneous to users.

Key Terms

TermMeaning
ThreadThe mechanism that runs a task
TaskThe actual job / work to be done
RunnableThe interface a task class must implement
Time SharingSingle CPU giving each thread a small time slice
👨‍🍳
Simple Analogy

Task (Runnable) = A recipe (the instructions). Thread = A chef (the one who executes the recipe). You can have multiple chefs running multiple recipes at once!

💡 Tips & Tricks — 30.2

Remember: A task is a Runnable object. A thread is the executor of that task. The task tells WHAT to do. The thread is HOW it gets done.

Exam Shortcut: "Runnable object" = instance of a class that implements the Runnable interface.

30.3

Creating Tasks and Threads

This is the most important section — the core of Java multithreading.

The 4-Step Pattern: T → T → S

  1. Create a Task Class (implements Runnable) 
    public class MyTask implements Runnable {
    @Override
    public void run() {
        // This is the task's job
        System.out.println("Task is running!");
    }
}

    Every task class must implement Runnable. The run() method contains what the task does.

  2. Create a Task Object 
    MyTask task = new MyTask();

    Just create an instance of your task class.

  3. Wrap It in a Thread 
    Thread thread = new Thread(task);

    Hand the task to a Thread object — like "hiring a chef for this recipe."

  4. Start the Thread 
    thread.start();

    Tells the JVM: "This thread is ready!" The JVM calls task.run() automatically in a new thread.

🔥 Full Example from the Book (Listing 30.1)

TaskThreadDemo.java 
public class TaskThreadDemo {
    public static void main(String[] args) {

        // Step 1: Create tasks
        Runnable printA   = new PrintChar('a', 100);
        Runnable printB   = new PrintChar('b', 100);
        Runnable print100 = new PrintNum(100);

        // Step 2: Create threads
        Thread thread1 = new Thread(printA);
        Thread thread2 = new Thread(printB);
        Thread thread3 = new Thread(print100);

        // Step 3: Start threads
        thread1.start();
        thread2.start();
        thread3.start();
    }
}

// Task 1 & 2: Print a character N times
class PrintChar implements Runnable {
    private char charToPrint;
    private int times;

    public PrintChar(char c, int t) {
        charToPrint = c;
        times = t;
    }

    @Override
    public void run() {
        for (int i = 0; i < times; i++) {
            System.out.print(charToPrint);
        }
    }
}

// Task 3: Print numbers 1 to N
class PrintNum implements Runnable {
    private int lastNum;

    public PrintNum(int n) { lastNum = n; }

    @Override
    public void run() {
        for (int i = 1; i <= lastNum; i++) {
            System.out.print(" " + i);
        }
    }
}

The 3 threads run concurrently, so output is mixed: aaab1b2aa3bba4b5a... — the exact order changes every run!

⚠️ Critical: start() vs run()

thread.start() ✅thread.run() ❌
Creates a new thread, runs task in itNo new thread! Runs in the CURRENT thread
True concurrency happensSequential execution — no concurrency
Always use thisCommon exam trap / mistake
💡 Tips & Tricks — 30.3

#1 Exam Mistake: Calling run() instead of start(). The code compiles fine but creates no new threads!

Memory Trick: T → T → S = Task → Thread → Start

Can you start a thread twice? NO. Calling start() on an already-started thread throws IllegalThreadStateException.

Design Principle: Keep the task (Runnable) separate from the thread (Thread). This is called "separation of concerns."

30.4

The Thread Class

The Thread class provides many methods to control how threads behave.

Key Methods Overview

start()
Starts the thread. JVM calls run() in a new thread.
isAlive()
Returns true if the thread is currently running.
setPriority(int p)
Sets thread priority from 1 (min) to 10 (max).
join()
Makes the calling thread wait for this thread to finish.
sleep(long millis)
Pauses the current thread for N milliseconds.
yield()
Temporarily gives up CPU time for other threads.
interrupt()
Interrupts a sleeping or waiting thread.
stop() / suspend() / resume()
DEPRECATED — Never use these.

Method 1: yield()

yield() = "I'll pause briefly; let other threads have a turn." Like a polite person saying "You go first."

public void run() {
    for (int i = 1; i <= lastNum; i++) {
        System.out.print(" " + i);
        Thread.yield();   // Gives other threads a chance
    }
}

Method 2: sleep(long millis)

sleep() = "This thread naps for X milliseconds." Must always be inside a try-catch!

✅ CORRECT — Loop INSIDE try-catch❌ INCORRECT — try-catch INSIDE loop
 
public void run() {
  try {
    while (...) {
      Thread.sleep(1000);
    }
  } catch (InterruptedException ex) {
    // Loop stops cleanly ✅
  }
}
 
public void run() {
  while (...) {
    try {
      Thread.sleep(1000);
    } catch (InterruptedException ex) {
      // Loop CONTINUES! Bug ❌
    }
  }
}

Method 3: join()

join() = "Wait for this thread to finish before I continue." Like waiting for a friend before starting dinner.

public void run() {
    Thread thread4 = new Thread(new PrintChar('c', 40));
    thread4.start();
    try {
        for (int i = 1; i <= lastNum; i++) {
            System.out.print(" " + i);
            if (i == 50) thread4.join(); // Wait for thread4 at i=50
        }
    } catch (InterruptedException ex) {}
}
// Numbers 1-50 print freely → wait for thread4 → then 51-100 print

Thread Priority

ConstantValueMeaning
Thread.MIN_PRIORITY1Lowest priority
Thread.NORM_PRIORITY5Default (normal)
Thread.MAX_PRIORITY10Highest priority
 
thread3.setPriority(Thread.MAX_PRIORITY); // thread3 gets more CPU time
⚠️ Starvation / Contention

If a high-priority thread runs forever, low-priority threads never get to run. Fix: high-priority threads should periodically call sleep() or yield() to give lower-priority threads a chance.

💡 Tips & Tricks — 30.4

Deprecated Trick: stop(), suspend(), resume() — all are deprecated. Remember: "SSS = deprecated".

sleep() vs yield(): Both are static methods. sleep() throws InterruptedException; yield() does NOT.

Priority Tip: Use constants (Thread.MAX_PRIORITY) not raw numbers — values may change in future Java versions.

Round Robin = when ALL threads have equal priority → each gets equal CPU time in turns.

30.6

Thread Pools

The Problem with Creating New Threads

If you have 1000 tasks, creating 1000 individual threads is memory-expensive, time-consuming, and hard to manage. Thread pools solve this.

🏊
What is a Thread Pool?

A collection of pre-created, reusable threads. A thread finishes its task → immediately picks up the next waiting task. No need to create a new thread each time!

// Thread Pool (3 threads)
Thread 1 ← picks up Task A, finishes, picks up Task D...
Thread 2 ← picks up Task B, finishes, picks up Task E...
Thread 3 ← picks up Task C, finishes, picks up Task F...

Task Queue: [A, B, C, D, E, F, ...]

The Executor Framework

Interface / ClassKey Methods
Executorexecute(Runnable task) — Submit a task
ExecutorServiceshutdown(), isShutdown(), isTerminated(), shutdownNow()
Executors (factory)newFixedThreadPool(N), newCachedThreadPool()

Creating Thread Pools

// Fixed pool: always exactly N threads
ExecutorService executor = Executors.newFixedThreadPool(3);

// Cached pool: creates threads as needed, reuses idle ones (60s timeout)
ExecutorService executor = Executors.newCachedThreadPool();

Full Example (Listing 30.3)

ExecutorDemo.java 
import java.util.concurrent.*;

public class ExecutorDemo {
    public static void main(String[] args) {

        // Step 1: Create a fixed thread pool with 3 threads
        ExecutorService executor = Executors.newFixedThreadPool(3);

        // Step 2: Submit tasks
        executor.execute(new PrintChar('a', 100));
        executor.execute(new PrintChar('b', 100));
        executor.execute(new PrintNum(100));

        // Step 3: Shut down (waits for running tasks to finish)
        executor.shutdown();
    }
}

Pool Size Comparison

Pool ConfigurationBehavior
newFixedThreadPool(3)3 tasks run concurrently at a time
newFixedThreadPool(1)Tasks run one by one (sequential)
newCachedThreadPool()All tasks run concurrently (new thread per task)
💡 Tips & Tricks — 30.6

Rule of Thumb: 1 task → use new Thread(task).start(). Multiple tasks → use a thread pool (ExecutorService).

shutdown() ≠ stop: shutdown() refuses new tasks but lets existing ones finish. Like "closing time" at a restaurant — no new customers, but seated ones are still served.

Exam Trick: Pool with 1 thread = sequential execution. Pool with N threads (N ≥ tasks) = all run concurrently.

Key Checkpoints — Questions & Solutions

Checkpoint 30.1
Why is multithreading needed? How can multiple threads run simultaneously in a single-processor system?
Answer

Why multithreading is needed:

  • To make programs more responsive and interactive
  • To improve performance by doing multiple things at once
  • Example: A word processor prints while you type, and auto-saves in the background
  • CPU is often idle (waiting for user input, disk I/O) — threads fill that idle time

How multiple threads run on a single CPU:

  • Through time sharing (time-slicing)
  • The OS rapidly switches between threads, giving each a small CPU slice
  • The switching is so fast it appears simultaneous to users
  • The OS handles all scheduling and resource allocation
Checkpoint 30.2
What is a runnable object? What is a thread?
Answer

Runnable object: An instance of a class that implements the Runnable interface. It represents a task — the actual work to be done. The run() method defines what the task does.

Runnable printA = new PrintChar('a', 100); // This is a runnable object

Thread: An object of the Thread class that provides the mechanism for executing a task — it is the "executor" that runs the Runnable.

Thread thread = new Thread(printA); // This is a thread

Analogy: Task (Runnable) = recipe. Thread = chef who follows the recipe.

Checkpoint 30.3
How do you define a task class? How do you create a thread for a task?
Answer

Defining a task class:

public class MyTask implements Runnable {
    @Override
    public void run() {
        // Define what this task does here
    }
}
  • The class must implement Runnable
  • Must override the run() method with the task logic

Creating a thread for the task (3 steps):

MyTask task = new MyTask();         // 1. Create task object
Thread thread = new Thread(task);   // 2. Wrap in a Thread
thread.start();                      // 3. Start → JVM calls run()
Checkpoint 30.4
What would happen if you replace the start() method with the run() method in lines 14–16 in Listing 30.1?
Answer

If thread1.start(), thread2.start(), thread3.start() are replaced with .run():

  • No new threads are created
  • run() executes directly in the main thread, one after another
  • Execution becomes sequential: first all 100 'a's print, then all 100 'b's, then 1–100
  • No interleaving — the output is always in fixed order
  • This defeats the entire purpose of multithreading
Key Lesson

start() creates a new thread and calls run() in it. run() alone just calls the method in the current thread — no new thread is created!

Checkpoint 30.5
What is wrong in the following two programs? Correct the errors.
Program (a) — Analysis
public class Test implements Runnable {
    public static void main(String[] args) { new Test(); }
    public Test() {
        Test task = new Test(); // ❌ BUG: creates Test() → calls constructor → creates Test() → ∞
        new Thread(task).start();
    }
    public void run() { System.out.println("test"); }
}

Problem: new Test() inside the constructor calls itself → infinite recursion → StackOverflowError.

Fix: Use this instead of creating a new Test():

public Test() {
    new Thread(this).start(); // ✅ Use 'this' — refers to current object
}
Program (b) — Analysis
public Test() {
    Thread t = new Thread(this);
    t.start();
    t.start(); // ❌ BUG: starting the same thread object TWICE
}

Problem: A thread can only be start()ed once. The second call throws IllegalThreadStateException at runtime.

Fix: Remove the second start(), or create two separate Thread objects:

public Test() {
    new Thread(this).start(); // First thread (new object)
    new Thread(this).start(); // Second thread (different object) ✅
}

Checkpoint 30.7
If a loop contains a method that throws an InterruptedException, why should the loop be placed inside a try-catch block?
Answer

The loop must be placed inside the try-catch (not the other way around):

✅ Correct — Loop INSIDE try-catch❌ Wrong — try-catch INSIDE loop
 
try {
  while (...) {
    Thread.sleep(1000);
  }
} catch (InterruptedException ex) {
  // Loop exits cleanly ✅
}
 
while (...) {
  try {
    Thread.sleep(1000);
  } catch (InterruptedException ex) {
    // Caught but loop CONTINUES ❌
  }
}

Explanation: If try-catch is inside the loop, the exception is caught but the loop keeps running — the thread continues executing even after interruption, which is almost always a bug. Wrapping the entire loop ensures the thread stops cleanly when interrupted.

Checkpoint 30.8
How do you set a priority for a thread? What is the default priority?
Answer

Setting priority:

thread.setPriority(Thread.MAX_PRIORITY);  // Sets to 10 ✅ Recommended
thread.setPriority(Thread.MIN_PRIORITY);  // Sets to 1
thread.setPriority(Thread.NORM_PRIORITY); // Sets to 5
thread.setPriority(7);                    // Raw number — not recommended
ConstantValue
Thread.MIN_PRIORITY1
Thread.NORM_PRIORITY5 (default)
Thread.MAX_PRIORITY10
  • Default priority: Thread.NORM_PRIORITY = 5
  • A thread inherits the priority of the thread that created it
  • Threads created by main also start at priority 5
  • JVM always picks the highest-priority runnable thread
  • Equal priority → round-robin scheduling
Checkpoint 30.14
What are the benefits of using a thread pool?
Answer
  1. Reusability: Threads are created once and reused — no overhead of repeatedly creating/destroying threads
  2. Performance: Starting a new thread for each task is expensive. Thread pools avoid this, making execution faster
  3. Resource Control: You can cap the maximum number of threads (e.g., newFixedThreadPool(10)) to prevent system overload
  4. Throughput: More tasks completed per unit time because threads aren't wasted being created and destroyed
  5. Simplicity: ExecutorService manages scheduling automatically — just submit tasks
  6. Prevents Crash: Without a pool, launching 10,000 threads could crash the JVM. A pool queues extra tasks and processes them orderly
Checkpoint 30.15
How do you create a thread pool with three fixed threads? How do you submit a task to a thread pool? How do you know that all the tasks are finished?
Answer

1. Create a fixed thread pool with 3 threads:

import java.util.concurrent.*;
ExecutorService executor = Executors.newFixedThreadPool(3);

2. Submit tasks to the pool:

executor.execute(new PrintChar('a', 100)); // Task 1
executor.execute(new PrintChar('b', 100)); // Task 2
executor.execute(new PrintNum(100));        // Task 3

3. Knowing when all tasks are finished:

executor.shutdown();              // Stop accepting new tasks

// Option A: Poll until done
while (!executor.isTerminated()) { // Loop until all done }
System.out.println("All tasks done!");

// Option B: Await with timeout
executor.awaitTermination(1, TimeUnit.HOURS);
Key Distinction

shutdown() = stops accepting new tasks, existing ones continue. isTerminated() = true only when ALL tasks have completed AND shutdown() was called first.

💡 TIP

Overall Tips & Tricks

🏆 Top 10 Exam Strategies

01
start() vs run()

The #1 trick question. start() creates a new thread. run() just calls a method in the current thread. NEVER confuse these.

02
Runnable over Thread subclass

Always prefer implementing Runnable over extending Thread. Better design and allows extending other classes.

03
sleep() needs try-catch

InterruptedException is a checked exception. Code with Thread.sleep() without try-catch won't compile.

04
Loop inside try-catch

For sleep() in a loop, the loop goes INSIDE the try-catch — not the try-catch inside the loop.

05
Thread pool for many tasks

1 task → single Thread. Many tasks → ExecutorService thread pool. Never create 1000 threads manually.

06
shutdown() ≠ stop

shutdown() gracefully waits for tasks to finish. stop() is deprecated and dangerous. Never use stop().

07
Priority ≠ guaranteed order

High priority = more likely to run first, but NOT guaranteed. Thread scheduling is OS-dependent.

08
Deprecated methods trap

stop(), suspend(), resume() = NEVER use. Common exam trap: code using these asking "what's wrong?"

09
One start() only

A thread can only be start()ed once. Calling it twice on the same object throws IllegalThreadStateException.

10
join() makes caller wait

If thread A calls threadB.join(), thread A pauses until thread B completes.
